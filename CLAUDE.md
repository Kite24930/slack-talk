## プロジェクト概要

**slack-talk** は、Slackメッセージのリアルタイム音声読み上げと、ウェイクワードによるハンズフリー音声送信を実現するmacOSネイティブアプリケーション。MacBook Pro M4 MAX (128GB) 単体で完結し、外部クラウドサービスは一切使用しない（Slack Socket Mode接続のみ）。

システム設計書: `docs/slack_tts_system_overview_v1.docx`

## アーキテクチャ

2つの主要データフロー:

1. **読み上げフロー**: Slackイベント(Socket Mode) → チャンネルフィルタ → FIFOキュー → TADA TTS(MPS加速) → macOSスピーカー出力
2. **音声送信フロー**: ウェイクワード検出(Porcupine) → Whisper STT(MPS加速) → Intent Parser(正規表現:「〇〇チャンネルに△△」) → Slack API投稿

### コンポーネント構成

| コンポーネント   | 役割                                   | 技術                        |
| ---------------- | -------------------------------------- | --------------------------- |
| Slack Listener   | Socket Modeでイベント受信・API送信     | slack-sdk (Python)          |
| Filter / Config  | チャンネル別読み上げON/OFF管理         | SQLite                      |
| Queue Manager    | FIFO読み上げ順序制御                   | asyncio.Queue               |
| TADA TTS Engine  | テキスト→音声変換（MPS加速）           | TADA 3B-ML (HumeAI OSS)     |
| Audio Output     | macOSスピーカーへ音声出力              | sounddevice                 |
| Wake Word Engine | バックグラウンドマイク常時監視         | Porcupine (Picovoice無料枠) |
| Whisper STT      | 音声→テキスト変換（MPS加速）           | Whisper large (OpenAI OSS)  |
| Intent Parser    | チャンネル名・メッセージ本文の分離解析 | 正規表現ルールベース        |

## 技術スタック

- **言語**: Python 3.11 + asyncio（非同期で全処理を統括）
- **Slack**: slack-sdk（Socket Modeで双方向リアルタイム通信）
- **TTS**: TADA 3B-ML — サンプルコードは`device='cuda'`前提のため`device='mps'`への書き換えが必要
- **STT**: Whisper large — `device='mps'`で動作
- **ウェイクワード**: Porcupine（無料枠はカスタムワード1件まで）
- **音声出力**: sounddevice（macOS Core Audio経由）
- **設定DB**: SQLite（sqlite3標準ライブラリ）
- **パッケージ管理**: Poetry または pip + venv

## MPS (Metal Performance Shaders) に関する注意

- TADAサンプルコードの`device='cuda'`を`device='mps'`に書き換える必要あり
- MPS未対応オペレーション発生時は`PYTORCH_ENABLE_MPS_FALLBACK=1`でCPUフォールバック
- 3Bモデル(bf16 約9GB)は128GBユニファイドメモリに十分収まる

## Claude Code 作業用ファイルの保存ルール

Claude Code のツールが生成する作業用ファイル（アプリケーション開発に直接関係しないもの）は `.artifacts/` 配下に保存すること。プロジェクトルートや `/tmp` に散在させない。

| 種別 | 保存先 |
|---|---|
| 設計書・計画書 | `docs/` |
| スクリーンショット | `.artifacts/screenshots/` |
| 生成HTML/図 | `.artifacts/generated/` |
| ダウンロード | `.artifacts/downloads/` |

- `docs/`, `.artifacts/` は `.gitignore` 対象（開発ツールの作業用であり、アプリケーションコードではない）
- スクリーンショット撮影時は必ず `.artifacts/screenshots/` をパスに指定する

## 開発ワークフロー（全タスク共通）

すべてのTaskは以下の流れで実施すること。例外なし。

### 1. 作業開始前
1. GitHub Issue を確認（なければ作成）
2. `main` から作業ブランチを切る（例: `feat/task-1-poetry-init`）
3. ブランチをリモートにpush

### 2. 作業中
- 区切りごとにこまめにコミットする（1つの大きなコミットにまとめない）
- コミットメッセージは変更内容を簡潔に記述
- **git関連メッセージ（Issue, PR, Commit等）はすべて日本語で記述する**

### 3. 作業完了後
1. **コントローラーによる独立検証**: `poetry run pytest -v` を実行し、全テストの通過を自ら確認する（サブエージェントの自己申告に依存しない）
2. サブエージェントによるレビュー（spec compliance + code quality）
3. 動作を伴う実装の場合は**主の確認を得る**（確認なしにマージしない）
4. レビュー通過後、最終コミット
5. PR作成 → mainへマージ
6. クリーンアップ:
   - リモート・ローカルの作業ブランチ削除
   - GitHub Issue をクローズ
   - ローカルを `main` に戻す
